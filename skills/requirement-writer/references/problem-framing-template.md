# Problem Framing Template

## Template structure

```markdown
# Problem Framing — {PROJECT_TITLE}

## Problem Statement

{Compose a fluent paragraph combining all six fields below.
Do NOT mechanically stitch fragments — write it as a natural,
readable problem statement that a stakeholder can understand
in 30 seconds.}

---

| Field | Description |
|-------|-------------|
| **WHO** | {target users + segmentation} |
| **WHAT Problem** | {specific pain points, current status quo, and how users cope today} |
| **WHEN** | {scenario, trigger, touchpoint} |
| **WHAT Job** | {core task the user needs to accomplish} |
| **WHAT benefits for the customer** | {tangible user value} |
| **WHAT benefits for the company** | {measurable business value} |
```

## Writing the Problem Statement paragraph

The statement sentence pattern is:

> WHO is facing WHAT problem, WHEN trying to complete WHAT job.
> Our solution should deliver WHAT benefits for the customer,
> and WHAT benefits for the company.

Adapt this pattern to sound natural in the document language.
The paragraph should read like a polished product brief, not a fill-in-the-blank exercise.

## Field guidelines

**WHO:** Include both the broad group and meaningful sub-segments.
Avoid generic descriptions like "all users".
> Example: "First-time visitors and infrequent returning users who are unfamiliar with the platform"

**WHAT Problem:** List concrete, observable issues. Use data or
specific behavioral patterns when available. Include the current
workaround if users have one.
> Example: "The homepage leads with promotions and product grids without first establishing what the platform offers; navigation mixes categories, scenarios, and trending searches; trust signals appear too late in the page"

**WHEN:** Be specific about the trigger moment — not just "when
using the app" but "when first landing on the homepage" or
"when switching between search results and product detail page".
> Example: "When landing on the English-language homepage for the first time"

**WHAT Job:** Frame as what the user is trying to achieve,
not what the product should do. Use Jobs-to-be-Done language.
> Example: "Quickly determine what the platform sells, whether it fits their needs, and find a starting point for browsing or purchasing"

**Benefits:** Customer benefits should be experiential (faster,
easier, more confident). Company benefits should be measurable
(conversion rate, retention, GMV).
> Example — customer: "Understand the platform's positioning faster and feel confident enough to start shopping"
> Example — company: "Increase new-visitor homepage conversion rate"
