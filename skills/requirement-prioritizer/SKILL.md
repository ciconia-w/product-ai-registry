# 需求优先级评估

## 五级优先级

| 等级 | 得分 | 含义 |
|------|------|------|
| P0 | 90-100 | 紧急且核心，立即安排 |
| P1 | 75-89 | 重要功能，近期优先 |
| P2 | 60-74 | 中等优先，正常排期 |
| P3 | 45-59 | 低优先，资源充裕时考虑 |
| P4 | 0-44 | 暂缓或暂不实现 |

## 六维度评分

| 维度 | 权重 | 反向 |
|------|------|------|
| 用户影响力 | 30% | 否 |
| 紧急程度 | 20% | 否 |
| 实现成本 | 15% | 是（成本越高分越低） |
| 战略价值 | 15% | 否 |
| 依赖关系 | 10% | 否 |
| 社区热度 | 10% | 否 |

评分标准详见 references/scoring-standards.md

## 工作流程

输入已分析的需求 → 六维度评分 → 加权计算总分 → 映射 P0-P4 → 输出结果

## 输出格式

```json
{
  "prioritized_at": "2026-03-23T14:30:00Z",
  "total_count": 1,
  "requirements": [{
    "priority": {
      "level": "P1",
      "score": 85,
      "scores": {"用户影响力": 9, "紧急程度": 7, ...},
      "reason": "影响大部分用户的重要功能"
    }
  }]
}
```

## 特殊情况

| 情况 | 处理 |
|------|------|
| 信息不足 | 缺失维度给5分，reason标注"信息不足" |
| Bug实际是需求 | 紧急程度给中等优先级 |
| 多个需求混合 | 评估整体影响，标注"含多个需求" |
