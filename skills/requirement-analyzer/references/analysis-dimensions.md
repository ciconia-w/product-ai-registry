# 分析维度详解

## 需求类型（4种）

| 类型 | 特征 | 关键词 |
|------|------|--------|
| 功能需求 | 新增或改进功能 | 添加、支持、希望、需要 |
| 非功能需求 | 性能、技术指标 | 性能、速度、流畅、响应时间 |
| 问题反馈 | Bug或功能失效 | 错误、无法、失败、崩溃、不工作 |
| 建议优化 | UI/UX改进 | 优化、改进、界面、体验 |

## 涉及模块

**判断优先级**: 输入数据中已包含 `module` 字段时，直接使用该值；否则根据内容推断。

**系统关键模块**: 系统设置、文件管理、任务栏、启动器、应用中心、终端、通知、截图、主题

**从内容推断规则**: 匹配关键词 → 选择权重最高模块 → 无匹配则"其他"

**已知的模块分类**（来自产品需求反馈系统）:
- 控制中心-网络、控制中心-帐户
- 文件管理器
- 系统安全
- 桌面插件（壁纸、屏保、桌面整理等）
- 更新管理系统
- 登录解锁
- 以及更多...

## 技术复杂度（3级）

| 等级 | 特征 | 示例 |
|------|------|------|
| 高 | 跨模块、架构变更、重构、技术集成 | 多模块联动、数据库改动 |
| 中 | 单模块功能、有一定挑战 | 新增功能、优化现有功能 |
| 低 | UI调整、配置修改、简单Bug | 改文字、改样式、修复小Bug |

## 用户场景（3要素）

```json
{
  "user_role": "用户角色（普通用户/管理员/开发者等）",
  "scenario": "在什么情况下、做什么操作",
  "expected_behavior": "希望看到或达成什么结果"
}
```

## 产品环境信息（4字段）

**重要**: 所有字段必须输出，无数据时为空字符串

| 字段 | 说明 | 提取方式 |
|------|------|----------|
| product_name | 所属产品名称 | 从需求内容或分类推断（如：UOS AI助手/UOS V20/UOS V25） |
| os | 操作系统 | 从需求内容提取（如：Deepin V20/Deepin V25/UOS） |
| os_version | 系统版本 | 从需求内容提取版本号（如：20/25/1070） |
| hardware_arch | 硬件架构 | 从需求内容提取（如：AMD64/ARM64/x86_64） |

## 核心价值

**要求**: 一句话概括需求的本质价值和用户受益

- 好: "提升文件操作的可预见性和用户体验"
- 差: "用户想要看到复制进度"

## 需求类型判断优先级

1. 明确的错误描述 → 问题反馈
2. 改进现有体验 → 建议优化
3. 性能相关 → 非功能需求
4. 其他 → 功能需求（默认）

## 分析质量自检

每次分析完成后确认:
- [ ] 每个需求都分配了类型
- [ ] 每个需求都匹配了模块（或标注"未知"）
- [ ] 每个需求都评估了复杂度
- [ ] 提炼了核心价值（一句话概括）
- [ ] 用户场景三要素完整（角色+场景+期望）
